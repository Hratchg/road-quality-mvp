<objective>
Materialize the Phase 7 dataset on disk and pause for operator hand-labeling. This plan has TWO parts:

**Part 1 (automated):** Run `fetch_eval_data.py --build --count 20` against the Phase 7 `_DEFAULT_LA_BBOXES` (Plan 07-02 deliverable) to download ~1500 LA images split 70/20/10 by sequence_id (Phase 2 D-09, Pitfall 7). Then run `scripts/prelabel.py` to populate auto-suggested YOLO labels using the public keremberke model.

**Part 2 (USER GATE — checkpoint):** Operator opens CVAT, imports the auto-labels, hand-corrects to clear ≥150 positive bboxes (≥30 in test split — SC #1), exports as Ultralytics YOLO format, and replaces the auto-labels with the corrected versions in `data/eval_la/labels/{train,val,test}/`.

Why a single plan with both parts:
- The pre-label step's output is the operator's INPUT to CVAT — semantically inseparable from the gate
- Splitting would create a Wave 2.5 dependency chain that adds zero parallelism
- The operator returns to the same plan after CVAT work to commit the labels and proceed

Phase pauses at the checkpoint. Plan 07-05 (training prep) is the next executable step once the operator returns.

Output: regenerated `data/eval_la/` (images NOT committed per `.gitignore`; labels committed after operator hand-correction).
</objective>
