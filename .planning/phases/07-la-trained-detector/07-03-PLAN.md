<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `--wipe-mapillary` CLI flag → DELETE statement | Operator-controlled flag, but the SQL literal is hard-coded; no operator-controlled WHERE clause possible |
| `wipe_mapillary_rows` ↔ DB | DELETE limited by `segment_defects.source = 'mapillary'` literal; CHECK constraint at the schema level bounds the value space |
| `--force-wipe` semantics | Operator can override the zero-rows latch but cannot extend the WHERE clause; risk is "delete with nothing to replace" not "delete unrelated rows" |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-04 | Tampering | `wipe_mapillary_rows` SQL | mitigate | Hard-coded WHERE clause `WHERE source = 'mapillary'` (no f-string, no parameterization). Verified by `test_wipe_mapillary_rows_uses_hardcoded_where` source-grep + the schema-level CHECK constraint at `db/migrations/002_mapillary_provenance.sql:30` which bounds `source` to literals. Even with a future typo on the literal, the CHECK constraint prevents matching unrelated rows. |
| T-07-03 | Race condition | wipe-then-INSERT | mitigate | Both wipe functions use the same `conn` as the subsequent `execute_values` INSERT (line 649); `conn.commit()` happens once per wipe + once at the end of INSERT. The whole sequence runs in a single transaction connection; failure between wipe and INSERT triggers `conn.rollback()` in the outer try/except (line 720). Brief zero-detections window only manifests if `--force-wipe` is passed to skip the safety latch — that's an explicit operator opt-in. |
| T-07-force-wipe-bypass | Tampering | `--force-wipe` flag | accept | Operator-explicit override of the safety latch. Documented in `--force-wipe` help text. Phase 7 prod re-ingestion (Plan 07-07) does NOT pass `--force-wipe`; it relies on detections being non-empty (training-zone bboxes have plenty of imagery). |
</threat_model>
