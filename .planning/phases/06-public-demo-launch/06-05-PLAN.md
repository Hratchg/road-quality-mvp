# Plan 06-05: Eval public baseline + pin HF revision

## Goal

Measure the public `keremberke/yolov8s-pothole-segmentation` model's
performance on the LA test split (Plan 06-04 hand-labels) and substitute
those numbers for the 14 TBD placeholders in `docs/DETECTOR_EVAL.md`.
Pin the model's HF revision SHA in `data_pipeline/detector_factory.py`
so production cannot drift to a future malicious upload.

This plan replaces the original "Plan 06-05: Train detector" after
D-09 deferred fine-tuning to Phase 7 (training would have been
underpowered with only 8 train bboxes; eval CIs would have been
embarrassing).

## Acceptance criteria

1. `python scripts/eval_detector.py --data data/eval_la/data.yaml
   --split test --json-out eval_results.json` exits 0
2. `eval_results.json` contains populated `precision_ci_95` and
   `recall_ci_95` 3-tuples (non-null, even if wide)
3. `docs/DETECTOR_EVAL.md` has zero remaining `TBD` placeholders;
   numbers come from `eval_results.json`
4. `docs/DETECTOR_EVAL.md` includes a new "Sample size caveat" section
   noting the 17-image / 3-positive test set and the implications for
   CI width
5. `data_pipeline/detector_factory._DEFAULT_HF_REPO` updated to
   `keremberke/yolov8s-pothole-segmentation@<sha>` where `<sha>` is
   the current HEAD revision SHA on the HF repo
6. Git commit references the model's pinned SHA in the message

## Tasks

1. **Run eval against test split** with the public model:
   ```
   /tmp/rq-venv/bin/python scripts/eval_detector.py \
     --data data/eval_la/data.yaml \
     --split test \
     --json-out /tmp/eval_results.json \
     -v
   ```
   Test split: 17 images, 3 positive bboxes (per Plan 06-04).
   Expected: precision/recall numbers with WIDE 95% CIs (e.g., recall
   could plausibly be `0.0–1.0` given 3 positives). That's expected;
   the caveat section explains why.

2. **Inspect output** — verify the JSON has `precision`, `recall`,
   `mAP_50`, `precision_ci_95`, `recall_ci_95`, and per-severity
   breakdown. If any field is null, debug before proceeding.

3. **Substitute TBDs in DETECTOR_EVAL.md.** Map each TBD to its
   corresponding eval_results.json field. Add a new section near the
   top, after the TL;DR:

   ```markdown
   ## Sample size caveat (Phase 6 baseline)

   The numbers below are computed against a 17-image / 3-positive-bbox
   LA test split (per Plan 06-04 of Phase 6). Confidence intervals are
   correspondingly wide — recall in particular has only 3 ground-truth
   positives, so a single false negative moves the point estimate by
   33%. These numbers are honest baselines, not production claims.

   Phase 7 will source ~10x more imagery, hand-label more aggressively,
   and replace these numbers with measurements from a fine-tuned
   LA-specific detector. The "previous baseline" section will preserve
   these numbers for traceability.
   ```

4. **Capture HF revision SHA** for the pin:
   ```
   /tmp/rq-venv/bin/python -c "
   from huggingface_hub import HfApi
   api = HfApi()
   info = api.model_info('keremberke/yolov8s-pothole-segmentation')
   print(info.sha)
   "
   ```

5. **Update `data_pipeline/detector_factory.py`:** change
   `_DEFAULT_HF_REPO = 'keremberke/yolov8s-pothole-segmentation'` to
   `_DEFAULT_HF_REPO = 'keremberke/yolov8s-pothole-segmentation@<sha>'`.
   Add a one-line comment referencing this plan + the rationale (Pitfall 8
   pickle-ACE drift protection).

6. **Verify the pinned model still loads:**
   ```
   /tmp/rq-venv/bin/python -c "
   import sys; sys.path.insert(0, '.')
   from data_pipeline import detector_factory
   d = detector_factory.get_detector()
   print('detector:', type(d).__name__)
   "
   ```

7. **Atomic commit** with `data_pipeline/detector_factory.py` +
   `docs/DETECTOR_EVAL.md` + `eval_results.json` (if conventional
   to commit; otherwise `.gitignore` it). Commit message references
   the SHA being pinned.
