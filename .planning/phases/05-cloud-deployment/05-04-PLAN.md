<objective>
Land the Fly artifacts for the frontend service: a multi-stage `deploy/frontend/Dockerfile` (node:20-alpine build stage → nginx:alpine runtime) that bakes `VITE_API_URL=https://road-quality-backend.fly.dev` into the JS bundle at build time, an `deploy/frontend/nginx.conf` with SPA fallback + smart cache headers, and a `deploy/frontend/fly.toml` that wires the inline `[build.args].VITE_API_URL` so deploys are reproducible. Plus a local docker-build smoke test that asserts the deployed-backend URL is baked into the bundle (SC #4 regression gate at the Docker layer).

Purpose: Without the build-arg discipline, a deploy would either bake an empty VITE_API_URL (the bundle hits "/" instead of the backend, every fetch 404s on the same nginx that's serving the frontend) OR bake `localhost:8000` (the bundle attempts to call the developer's machine, which is unreachable from a browser on the public internet). RESEARCH Pitfall 4 documents both failure modes; this plan's verify gates close them. Without the SPA fallback in nginx.conf, a hard refresh on /route would 404 (nginx can't find /route on disk; only client-side react-router knows /route is a route). Without cache headers, every page load re-downloads the entire bundle.

Output: 4 new files (1 Dockerfile + 1 nginx config + 1 fly.toml + 1 verification shell script). All under `deploy/frontend/`. No code changes; no test changes; no docker-compose.yml or frontend/src/api.ts modifications. PATTERNS confirmed `frontend/src/api.ts:3` already reads `import.meta.env.VITE_API_URL` correctly — the runtime contract is shipped; this plan ships only the build pipeline.

<!-- RESEARCH §2 lines 311-336 verbatim deploy/frontend/nginx.conf (the implementation target) -->

```nginx
server {
    listen 80 default_server;
    listen [::]:80 default_server;

    root /usr/share/nginx/html;
    index index.html;

    # SPA fallback — react-router-dom routes resolve client-side
    location / {
        try_files $uri $uri/ /index.html;
    }

    # Cache static assets aggressively (Vite hashes filenames)
    location ~* \.(js|css|woff2?|png|jpg|svg)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # Don't cache index.html — it points to the latest hashed bundles
    location = /index.html {
        add_header Cache-Control "no-cache, must-revalidate";
    }
}
```

**Step B: Create `deploy/frontend/Dockerfile`** with the RESEARCH §2 verbatim content. Final file:

```dockerfile
# deploy/frontend/Dockerfile — multi-stage Vite build → nginx:alpine static
# serve for the road-quality-frontend Fly app (Phase 5, REQ-prod-deploy
# SC #1 + SC #4, CONTEXT D-03).
#
# Stage 1 (build): node:20-alpine runs `npm ci && npm run build`. The
# VITE_API_URL build ARG is propagated as an ENV var BEFORE `npm run build`
# so Vite's import.meta.env.VITE_API_URL resolves at compile time and the
# literal "https://road-quality-backend.fly.dev" is baked into the JS bundle.
# Runtime env vars (docker run -e VITE_API_URL=...) DO NOT reach the bundle —
# this is a Vite design choice, NOT a Docker quirk (RESEARCH Pitfall 4).
#
# Stage 2 (runtime): nginx:alpine serves the built dist/ output. ~50 MB
# image; nginx handles SPA fallback, gzip, and cache headers via the
# baked-in deploy/frontend/nginx.conf.
#
# Build context is the repo root (matches deploy/db/Dockerfile's convention):
# `flyctl deploy --config deploy/frontend/fly.toml .` from repo root resolves
# all COPY paths.

# Stage 1: build the Vite bundle
FROM node:20-alpine AS build
WORKDIR /app

# Copy package manifests first for Docker layer caching: as long as
# package.json + package-lock.json don't change, the npm ci layer is reused
# across rebuilds (saves ~30-60s per local rebuild iteration).
COPY frontend/package.json frontend/package-lock.json ./
RUN npm ci

# Copy the rest of the frontend source.
COPY frontend/ ./

# VITE_API_URL is consumed by import.meta.env.VITE_API_URL at BUILD TIME.
# Runtime env vars (docker run -e VITE_API_URL=...) DO NOT propagate into
# the bundle. SC #4 regression gate (Plan 05-04 Task 4 + Plan 05-05 CI):
# `grep -r road-quality-backend.fly.dev /usr/share/nginx/html/assets/`
# must return matches in the deployed image; `grep -r localhost
# /usr/share/nginx/html/assets/` must return ZERO matches.
ARG VITE_API_URL
ENV VITE_API_URL=$VITE_API_URL

RUN npm run build  # produces /app/dist with index.html + hashed assets

# Stage 2: serve the static bundle via nginx
FROM nginx:alpine

# Replace nginx's default config with our SPA-aware one.
COPY deploy/frontend/nginx.conf /etc/nginx/conf.d/default.conf

# Copy the built bundle from Stage 1.
COPY --from=build /app/dist /usr/share/nginx/html

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

Critical:
1. `node:20-alpine` (NOT `node:20-slim`) — RESEARCH §2 line 285 picks alpine for the smaller build image footprint. PATTERNS P-9 line 202 used `node:20-slim` as a placeholder; RESEARCH's alpine choice supersedes (alpine images are ~40% smaller and the build phase doesn't need glibc-specific binaries).
2. `nginx:alpine` (NOT just `nginx:latest`) — alpine variant is ~50 MB vs ~140 MB for the default nginx image.
3. `ARG VITE_API_URL` is declared INSIDE the build stage (not before `FROM`); Docker's ARG scoping means a stage-level ARG must be re-declared after each FROM. We don't propagate the ARG to stage 2 (nginx) because nginx doesn't read VITE_API_URL — it only serves the already-built bundle.
4. `ENV VITE_API_URL=$VITE_API_URL` — converts the build-arg into an env var visible to `npm run build`. Without the ENV line, `npm run build` wouldn't see the ARG (Docker ARGs are NOT available to RUN commands as env vars by default in scoped form; the ENV line bridges them).
5. `COPY frontend/package.json frontend/package-lock.json ./` is a SEPARATE layer from `COPY frontend/ ./` — this is the standard Docker caching trick for npm-based builds. As long as package*.json files don't change, the npm ci layer (which can take 30-60s) is reused across rebuilds.
6. `COPY deploy/frontend/nginx.conf /etc/nginx/conf.d/default.conf` — overwrites nginx's default config with the SPA-aware one from Task 2.
7. `EXPOSE 80` is documentation (Fly's [http_service].internal_port=80 is what actually maps); `CMD ["nginx", "-g", "daemon off;"]` is the standard nginx-in-Docker pattern (without `daemon off`, nginx forks and the container exits immediately).

**Step C: Create `deploy/frontend/test-build.sh`** — a local smoke test that builds the image with a test VITE_API_URL and asserts the URL is baked into the JS bundle. Final file:

```bash
#!/usr/bin/env bash
# deploy/frontend/test-build.sh — Local smoke test for deploy/frontend/Dockerfile.
#
# Builds the image with VITE_API_URL=https://road-quality-backend.fly.dev
# (the production target), then runs the image transiently and asserts:
#   1. The built JS bundle contains the literal string "road-quality-backend.fly.dev"
#   2. The built JS bundle does NOT contain "localhost" or "127.0.0.1"
#
# This is the SC #4 regression gate at the Docker layer. Plan 05-05's GH
# Actions workflow may invoke this same script as a pre-deploy CI check.
#
# Usage: from repo root, run: bash deploy/frontend/test-build.sh
# Exit codes: 0 = build successful + URL baked in; non-zero = failure.

set -euo pipefail

if [ ! -f "deploy/frontend/Dockerfile" ]; then
    echo "ERROR: run this from the repo root (deploy/frontend/Dockerfile not found)" >&2
    exit 1
fi

IMAGE_TAG="road-quality-frontend:test"
TARGET_URL="https://road-quality-backend.fly.dev"

echo "Building $IMAGE_TAG with VITE_API_URL=$TARGET_URL ..."
docker build \
    -f deploy/frontend/Dockerfile \
    --build-arg "VITE_API_URL=$TARGET_URL" \
    -t "$IMAGE_TAG" \
    .

echo "Asserting $TARGET_URL is baked into the JS bundle..."
if ! docker run --rm "$IMAGE_TAG" sh -c \
    "grep -rq 'road-quality-backend.fly.dev' /usr/share/nginx/html/assets/"; then
    echo "FAIL: SC #4 violation — VITE_API_URL was NOT baked into the bundle" >&2
    echo "  Vite consumes import.meta.env.VITE_API_URL at BUILD time, not runtime." >&2
    echo "  Re-check that ARG VITE_API_URL + ENV VITE_API_URL=\$VITE_API_URL appear" >&2
    echo "  BEFORE 'RUN npm run build' in deploy/frontend/Dockerfile." >&2
    exit 1
fi
echo "  PASS: $TARGET_URL found in /usr/share/nginx/html/assets/"

echo "Asserting localhost is NOT baked into the JS bundle (SC #4 negative half)..."
if docker run --rm "$IMAGE_TAG" sh -c \
    "grep -rq 'localhost' /usr/share/nginx/html/assets/" 2>/dev/null; then
    echo "FAIL: SC #4 violation — localhost was found in the production bundle" >&2
    echo "  Confirm VITE_API_URL was passed via --build-arg and did not fall through" >&2
    echo "  to api.ts's '|| \"/api\"' default." >&2
    exit 1
fi
echo "  PASS: no 'localhost' in production bundle"

echo "Asserting nginx config is in place..."
if ! docker run --rm "$IMAGE_TAG" test -f /etc/nginx/conf.d/default.conf; then
    echo "FAIL: /etc/nginx/conf.d/default.conf not present in image" >&2
    exit 1
fi
echo "  PASS: nginx config baked in"

echo "Asserting index.html is in place..."
if ! docker run --rm "$IMAGE_TAG" test -f /usr/share/nginx/html/index.html; then
    echo "FAIL: /usr/share/nginx/html/index.html not present" >&2
    exit 1
fi
echo "  PASS: index.html baked in"

echo
echo "PASS: deploy/frontend/Dockerfile builds successfully; SC #4 regression gate green."
```

```nginx
# deploy/frontend/nginx.conf — nginx server block for the road-quality-frontend
# Fly app (Phase 5, REQ-prod-deploy SC #1).
#
# Three concerns:
#   1. SPA fallback: react-router-dom v7 routes (e.g., /route, /map) resolve
#      client-side. nginx must serve /index.html for any path that doesn't
#      match a real file on disk, otherwise hard-refresh on /route returns 404.
#   2. Aggressive caching for hashed assets: Vite hashes filenames
#      (assets/index-<hash>.js), so any URL change implies new content.
#      Cache-Control: public, immutable + 1y expiry is correct.
#   3. No caching for index.html: it points to the latest hashed bundles via
#      <script src="assets/index-<hash>.js">; if cached, users see stale bundles.

server {
    listen 80 default_server;
    listen [::]:80 default_server;

    root /usr/share/nginx/html;
    index index.html;

    # SPA fallback — react-router-dom routes resolve client-side.
    # try_files: serve $uri (exact file match), then $uri/ (directory),
    # then /index.html (the SPA's entry point) so client-side routing fires.
    location / {
        try_files $uri $uri/ /index.html;
    }

    # Cache static assets aggressively — Vite hashes filenames, so any URL
    # change implies new content. immutable tells clients "never re-validate
    # this URL" (saves a HEAD request on every page load).
    location ~* \.(js|css|woff2?|png|jpg|svg)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # Don't cache index.html — it points to the latest hashed bundles.
    # If we cached, deploys wouldn't take effect until the cached index.html
    # expired (could be days for users with stale browser caches).
    location = /index.html {
        add_header Cache-Control "no-cache, must-revalidate";
    }
}
```

Critical:
1. `listen 80 default_server;` + `listen [::]:80 default_server;` — IPv4 + IPv6 (Fly's edge proxy speaks both). The `default_server` flag makes this block the catch-all for any Host header that doesn't match a more specific server block (we have none, so this is the only block).
2. `root /usr/share/nginx/html;` — matches the COPY destination in the Dockerfile (`COPY --from=build /app/dist /usr/share/nginx/html`).
3. `try_files $uri $uri/ /index.html;` — the canonical SPA fallback. nginx tries the literal path, then path-as-directory (with index.html), then falls through to /index.html for any unknown path. RESEARCH A9 confirms this works for react-router-dom v7.
4. The `\.(js|css|woff2?|png|jpg|svg)$` regex — covers Vite's typical asset types. Adds `woff2` (web fonts) as well as `woff` (older browsers). The `?` makes `woff` optional so `woff2` matches too.

Both work together; `Cache-Control` overrides `Expires` for clients that respect the newer header.
6. `location = /index.html { ... }` — exact-match (`=` modifier) on `/index.html`. The `Cache-Control: no-cache, must-revalidate` tells clients "you may store this, but always revalidate before using" — so deploys take effect on the next page load.
7. NO `add_header X-Frame-Options "SAMEORIGIN";` or other security headers in this block — Fly's edge proxy may add some by default; the project has no specific clickjacking concern at M1 scale. Phase 6+ may add a security-headers block if a public demo reveals the need.
