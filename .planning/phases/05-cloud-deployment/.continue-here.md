# BLOCKING CONSTRAINTS — Read Before Anything Else

> Each constraint below was discovered through actual failure this session.
> Acknowledge each one explicitly before proceeding.

- [ ] CONSTRAINT: **Never run `pgr_createTopology` through `flyctl proxy`**. The wireguard tunnel cannot sustain the long-running query. When the proxy connection dies mid-query, PostgreSQL aborts the transaction and tries to roll back, which combined with WAL pressure crashes the DB into a recovery loop. Run it via `flyctl ssh console -C "psql ..."` directly on the db machine.

**Do not proceed with topology creation until this is acknowledged.**

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| pgr_createTopology via proxy | `flyctl proxy 15432:5432` + `psql ... pgr_createTopology(...)` from host. The proxy times out on the multi-minute query, postgres aborts, rollback writes WAL, hits volume capacity, FATAL: No space left on device → recovery crash loop | blocking | Always run `pgr_createTopology` (and any long DDL) via `flyctl ssh console -a road-quality-db -C "psql -U rq -d roadquality -c \"...\""`. Direct on-machine execution avoids the wireguard timeout entirely. |
| Trusting Phase 5 RESEARCH/CONTEXT memory sizing | The 512MB DB machine + 1GB volume defaults were sized for an empty schema. After seeding 200k segments + 125k defects + scores + indexes + WAL, both ran out simultaneously. | advisory | Phase 5 plan (or a Phase 5.1 polish plan) should require ≥2GB RAM and ≥3GB volume for production deploys with a real seed.
