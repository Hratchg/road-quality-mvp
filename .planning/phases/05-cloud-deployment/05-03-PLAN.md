**Step B: Create `deploy/db/Dockerfile`** with the exact RESEARCH §1 + Migration Init Flow content. The file:

```dockerfile
# deploy/db/Dockerfile — Postgres 16 + PostGIS 3.4 + pgRouting 3.8 image for
# the road-quality-db Fly app (Phase 5, REQ-prod-deploy SC #1).
#
# Built from postgis/postgis:16-3.4 (verified Debian 11 bullseye base via
# `docker run --rm postgis/postgis:16-3.4 cat /etc/os-release`); apt-installs
# postgresql-16-pgrouting from the image's pre-configured bullseye-pgdg repo
# (verified to ship pgRouting 3.8.0 via curl + gunzip on the apt Packages
# manifest as of 2026-04-27 — RESEARCH §1 line 102).
#
# Migrations are baked INTO the image so a fresh Fly volume picks them up
# automatically on first boot via Postgres' /docker-entrypoint-initdb.d/
# auto-init flow. Idempotency: all migrations use CREATE TABLE IF NOT EXISTS
# / CREATE INDEX IF NOT EXISTS, so manual re-application via flyctl ssh is
# also safe (RESEARCH Migration Init Flow + Pitfall 6 covers volume-restore
# corner case).
#
# Baking strategy chosen over Fly file-mounts (RESEARCH alternatives table:
# "B) Mount db/migrations/ via Fly [mounts]" rejected) because the image IS
# the deploy artifact — no drift between volume snapshot and migration set.

FROM postgis/postgis:16-3.4

RUN apt-get update \
 && apt-get install -y --no-install-recommends postgresql-16-pgrouting \
 && rm -rf /var/lib/apt/lists/*

# Init scripts run in lexicographic order on FIRST boot of an EMPTY $PGDATA
# volume. Numeric prefixes match docker-compose.yml's local-dev convention
# (one-prefix-lower than the compose mounts so future Phase 6+ migrations
# can slot in as 04-*.sql etc.).
COPY db/init-pgrouting.sh                      /docker-entrypoint-initdb.d/00-init-pgrouting.sh
COPY db/migrations/001_initial.sql             /docker-entrypoint-initdb.d/01-schema.sql
COPY db/migrations/002_mapillary_provenance.sql /docker-entrypoint-initdb.d/02-mapillary.sql
COPY db/migrations/003_users.sql               /docker-entrypoint-initdb.d/03-users.sql
```

Critical:
- Build context MUST be the repo root, NOT `deploy/db/`. The COPY paths use `db/migrations/...` and `db/init-pgrouting.sh` relative to the repo root. The Fly deploy command in Plan 05-05 will run `flyctl deploy --config deploy/db/fly.toml .` from repo root, which sets the build context to `.` (the repo root).
- Each `COPY` command targets ONE specific file with ONE specific destination filename. Avoid `COPY db/migrations/*.sql /docker-entrypoint-initdb.d/` (the unprefixed glob form) because the destination filenames would lose their numeric prefix and run in alphabetic order based on the source filename — which happens to be the same lexicographic order in our case (001/002/003), but is fragile if Phase 6+ adds a `00X_pre_initial.sql` that needs to come BEFORE `001_initial.sql`. Explicit file-by-file COPY is safer.
- The `00-init-pgrouting.sh` SHELL script must remain executable in the image. Postgres' init-script runner detects `*.sh` files and runs them via `bash`; `*.sql` files run via `psql`. The chmod is preserved by COPY's default mode-preservation (the file's mode in the host filesystem propagates).

```bash
#!/usr/bin/env bash
# deploy/db/test-build.sh — Local smoke test for deploy/db/Dockerfile.
#
# Builds the image from repo root and asserts that pgRouting 3.8 (or newer)
# is installed in the resulting image. RESEARCH §1 line 138-146 verbatim.
#
# Usage: from repo root, run: bash deploy/db/test-build.sh
# Exit codes: 0 = build successful + pgRouting present; non-zero = failure.

set -euo pipefail

# Ensure we're at repo root (the build context).
if [ ! -f "deploy/db/Dockerfile" ]; then
    echo "ERROR: run this from the repo root (deploy/db/Dockerfile not found in cwd)" >&2
    exit 1
fi

IMAGE_TAG="road-quality-db:test"

echo "Building $IMAGE_TAG from repo root..."
docker build -f deploy/db/Dockerfile -t "$IMAGE_TAG" .

echo "Verifying pgRouting extension is installed..."
PGROUTING_FILE=$(docker run --rm "$IMAGE_TAG" sh -c \
    "ls /usr/share/postgresql/16/extension/pgrouting--*.sql 2>/dev/null | grep -v -- '--.*--' | head -1" \
    || true)

if [ -z "$PGROUTING_FILE" ]; then
    echo "FAIL: pgrouting--*.sql not found in image" >&2
    exit 1
fi

echo "  Found: $PGROUTING_FILE"

# Extract version: pgrouting--3.8.0.sql -> 3.8.0
VERSION=$(echo "$PGROUTING_FILE" | sed -E 's|.*pgrouting--([0-9.]+)\.sql|\1|')
echo "  pgRouting version: $VERSION"

# Assert version is >= 3.6 (project floor per CON-stack-database).
MAJOR=$(echo "$VERSION" | cut -d. -f1)
MINOR=$(echo "$VERSION" | cut -d. -f2)
if [ "$MAJOR" -lt 3 ] || { [ "$MAJOR" -eq 3 ] && [ "$MINOR" -lt 6 ]; }; then
    echo "FAIL: pgRouting $VERSION is below project floor 3.6" >&2
    exit 1
fi

echo "Verifying baked migrations are present..."
for migration in 00-init-pgrouting.sh 01-schema.sql 02-mapillary.sql 03-users.sql; do
    if ! docker run --rm "$IMAGE_TAG" test -f "/docker-entrypoint-initdb.d/$migration"; then
        echo "FAIL: /docker-entrypoint-initdb.d/$migration missing from image" >&2
        exit 1
    fi
    echo "  Present: /docker-entrypoint-initdb.d/$migration"
done

echo
echo "PASS: deploy/db/Dockerfile builds successfully; pgRouting $VERSION installed; all 4 init scripts baked in."
```

**Create `deploy/db/fly.toml`** with the RESEARCH §2 verbatim skeleton. Final file:

```toml
# deploy/db/fly.toml — Fly app config for road-quality-db (Phase 5,
# REQ-prod-deploy SC #1, CONTEXT D-02).
#
# Single-primary Postgres + PostGIS + pgRouting backed by a Fly volume.
# Internal-only — no [http_service]; reachable from other Fly apps via
# road-quality-db.internal:5432 (basic IPv6 DNS over WireGuard-encrypted
# Fly private networking; sslmode=require NOT needed per RESEARCH §2 line 364).
#
# Volume snapshots: Fly enables daily snapshots with 5-day retention by
# default (RESEARCH A6); no extra config needed for M1 demo backup posture.

app = "road-quality-db"
primary_region = "lax"  # closest to LA dataset center (34.0522, -118.2437)

[build]
  dockerfile = "deploy/db/Dockerfile"

[env]
  POSTGRES_DB = "roadquality"
  POSTGRES_USER = "rq"
  # POSTGRES_PASSWORD set via `fly secrets set --app road-quality-db
  # POSTGRES_PASSWORD=<value>` — NEVER committed to git per CONTEXT D-05.

[[mounts]]
  source = "pgdata"
  destination = "/var/lib/postgresql/data"
  initial_size = "1gb"
  # Fly snapshots run daily (default 5-day retention) — no extra config.

# Internal-only — no [http_service] block, no public TCP handlers.
# Other Fly apps reach this at road-quality-db.internal:5432.
[[services]]
  internal_port = 5432
  protocol = "tcp"
  auto_stop_machines = false  # DB must stay up — never auto-stop
  auto_start_machines = false
  min_machines_running = 1

  [[services.tcp_checks]]
    grace_period = "30s"
    interval = "15s"
    timeout = "5s"

[[vm]]
  size = "shared-cpu-1x"
  memory = "512mb"
```

Key invariants:
1. `app = "road-quality-db"` — locked name; cross-references the DATABASE_URL secret in deploy/backend (`postgres://rq:${PG_PASSWORD}@road-quality-db.internal:5432/roadquality`).
2. `primary_region = "lax"` — locked to LA per the dataset center; multi-region is out of scope for M1.
3. `[build].dockerfile = "deploy/db/Dockerfile"` — Fly resolves this RELATIVE to where flyctl deploy runs from. The GH Actions workflow in Plan 05-05 runs from repo root, so this path resolves to `<repo_root>/deploy/db/Dockerfile`.
